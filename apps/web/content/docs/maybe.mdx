The `Maybe` type represents an optional value—it can either contain a value or be empty. It's perfect for handling values that might be null or undefined, but in a type-safe way.

## The Problem with null and undefined

How many times have you seen (or written) code like this?

```typescript
interface User {
  name: string;
  email: string;
  phone?: string;  // Optional!
}

function getPhone(user: User): string {
  // What if phone is undefined?
  return user.phone.toUpperCase(); // 💥 Runtime error!
}
```

The `?` makes `phone` optional at the type level, but it doesn't help you handle the "missing" case. You still have to remember to check for undefined everywhere.

<Callout type="warning">
  This is the "billion-dollar mistake" as Tony Hoare called it. null and undefined force you to constantly check for missing values.
</Callout>

## Enter Maybe: Explicit Optional Values

Maybe makes optionality **explicit** in your type signature:

```typescript
import { some, none, isSome, isNone } from "@deessejs/core";

// A value that might exist
const userPhone: Maybe<string> = some("+1-555-0123");
const noPhone: Maybe<string> = none();

// TypeScript forces you to handle both cases
if (isSome(userPhone)) {
  console.log(userPhone.value.toUpperCase()); // "+1-555-0123"
}
```

## Imperative vs Functional

Here's how you'd traditionally handle optional values:

<Tabs items={['Traditional', 'With Maybe']}>
<Tab value="Traditional">
```typescript
interface Config {
  debug?: boolean;
  logLevel?: "info" | "warn" | "error";
}

function processConfig(config: Record<string, unknown>): Config {
  return {
    debug: config.debug !== undefined ? config.debug : false,
    logLevel: config.logLevel !== undefined ? config.logLevel : "info",
  };
}
```
</Tab>
<Tab value="With Maybe">
```typescript
import { some, none, fromNullable, map, getOrElse } from "@deessejs/core";

interface Config {
  debug: boolean;
  logLevel: "info" | "warn" | "error";
}

function processConfig(config: Record<string, unknown>): Config {
  return {
    debug: fromNullable(config.debug).getOrElse(false),
    logLevel: fromNullable(config.logLevel).getOrElse("info"),
  };
}
```
</Tab>
</Tabs>

The intent is now **explicit**: "use this value, or fall back to this default."

<Callout>
  Notice how `fromNullable(config.debug).getOrElse(false)` clearly communicates: "try to get debug, but default to false if it's missing."
</Callout>

## Creating Maybe

### some - Creating a Present Value

When you know the value exists:

```typescript
import { some, isSome } from "@deessejs/core";

const present: Maybe<number> = some(42);
console.log(isSome(present)); // true
console.log(present.value);    // 42
```

### none - Creating an Absent Value

When the value doesn't exist:

```typescript
import { none, isNone } from "@deessejs/core";

const absent: Maybe<number> = none();
console.log(isNone(absent)); // true
```

### fromNullable - From null/undefined

Convert an existing nullable value:

```typescript
import { fromNullable } from "@deessejs/core";

fromNullable(42);          // Some(42)
fromNullable(null);        // None
fromNullable(undefined);   // None
```

This is incredibly useful for wrapping external APIs:

```typescript
import { fromNullable } from "@deessejs/core";

interface Response {
  data?: { name: string };
}

function getName(response: Response): Maybe<string> {
  return fromNullable(response.data?.name);
}
```

## Transforming Maybe

### map - Transform the Value

When you want to transform the value inside the Maybe:

```typescript
import { some, none, map } from "@deessejs/core";

some(10)
  .map(x => x * 2)   // Some(20)
  .map(x => x + 1); // Some(21)

none()
  .map(x => x * 2); // None - transformation is skipped
```

<Callout>
  Think of `map` as an assembly line: the value passes through transformations, but if there's no value (None), the transformations are simply skipped.
</Callout>

### flatMap - Chain Operations That Might Fail

When your transformation returns a Maybe:

```typescript
import { some, flatMap, fromNullable } from "@deessejs/core";

interface User {
  address?: { city: string };
}

const getCity = (user: User): Maybe<string> =>
  fromNullable(user.address?.city);

// Without flatMap - nested Maybes!
const badResult = some({ address: { city: "Paris" } })
  .map(user => getCity(user));
// Type: Maybe<Maybe<string>> 😱

// With flatMap - clean!
const result = some({ address: { city: "Paris" } })
  .flatMap(user => getCity(user));
// Type: Maybe<string> ✨
```

## Extracting the Value

### getOrElse - Default Value

For simple cases where you want a default:

```typescript
import { some, none, getOrElse } from "@deessejs/core";

some(10).getOrElse(0); // 10
none().getOrElse(0);    // 0
```

This is perfect for configuration with sensible defaults:

```typescript
const timeout = fromNullable(config.timeout)
  .getOrElse(5000);
```

### getOrCompute - Lazy Default

When computing the default is expensive:

```typescript
import { none, getOrCompute } from "@deessejs/core";

const result = none()
  .getOrCompute(() => {
    console.log("Computing expensive default...");
    return 42;
  });
// Logs: "Computing expensive default..."
```

### match - Pattern Matching

For exhaustive handling that forces you to cover both cases:

```typescript
import { some, none, match } from "@deessejs/core";

const message = some(42)
  .match(
    (value) => `Got: ${value}`,
    () => "No value"
  );
// "Got: 42"

const emptyMessage = none()
  .match(
    (value) => `Got: ${value}`,
    () => "No value"
  );
// "No value"
```

<Callout type="warning">
  Why is this powerful? TypeScript will warn you if you add a new variant to Maybe but forget to handle it!
</Callout>

### tap - Side Effects

When you want to do something with the value without changing it:

```typescript
import { some, tap } from "@deessejs/core";

some(42)
  .tap(value => console.log("Got value:", value))
  .tap(value => console.log("Doubled:", value * 2));
// Logs: "Got value: 42"
// Logs: "Doubled: 84"
```

This is great for logging or debugging without disrupting your data flow.

## Type Guards

### isSome / isNone

Check what's inside without extracting:

```typescript
import { some, none, isSome, isNone } from "@deessejs/core";

const value: Maybe<number> = some(42);

if (isSome(value)) {
  // TypeScript knows value is Some<number>
  console.log(value.value); // 42
}

if (isNone(value)) {
  // TypeScript knows value is None
  console.log("No value");
}
```

## Unit Value

### someUnit - A Some with Undefined

Sometimes you just need to know something exists, not what it is:

```typescript
import { someUnit, isSome } from "@deessejs/core";

const hasPermission = someUnit();
console.log(isPermission(hasPermission)); // true
console.log(hasPermission.value);        // undefined
```

## When to Use Maybe

Use Maybe when:

- A value might or might not exist
- You're dealing with optional properties
- You want to make the "missing" case explicit in your API
- You want TypeScript to force you to handle both cases
