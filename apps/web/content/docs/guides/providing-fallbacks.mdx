Sometimes an operation fails but you want to continue with a default value instead of propagating the error. Here's how to handle that gracefully.

## The Problem

Traditional fallback handling is verbose:

```typescript
function getConfig(key: string): string {
  const value = config[key];

  if (value === undefined) {
    return "default_value";
  }

  return value;
}
```

And for more complex cases:

```typescript
let user: User | null = null;

try {
  user = fetchUser(id);
} catch {
  // What now?
  user = getDefaultUser();
}

if (user === null) {
  user = getDefaultUser();
}
```

Problems:
- Inconsistent patterns for defaults
- Easy to forget to handle the null case
- No type safety on what defaults are

## The Solution: getOrElse

### Basic Usage

```typescript
import { ok, err, getOrElse, Maybe, some, none } from "@deessejs/core";

// With Result
const result = ok(42).getOrElse(0);   // 42
const failed = err("error").getOrElse(0); // 0

// With Maybe
const present = some(42).getOrElse(0); // 42
const absent = none().getOrElse(0);    // 0
```

### Real-World Example: Configuration

```typescript
import { fromNullable, getOrElse } from "@deessejs/core";

interface Config {
  apiUrl: string;
  timeout: number;
  retries: number;
}

const getConfig = (env: Record<string, string | undefined>): Config => ({
  apiUrl: fromNullable(env.API_URL)
    .getOrElse("https://api.example.com"),

  timeout: fromNullable(env.TIMEOUT)
    .map(Number)
    .getOrElse(5000),

  retries: fromNullable(env.RETRIES)
    .map(Number)
    .getOrElse(3),
});
```

<Callout>
  Notice how `fromNullable(env.TIMEOUT).map(Number).getOrElse(5000)` clearly expresses: "Try to get timeout, convert to number, default to 5000 if missing."
</Callout>

## Lazy Defaults: getOrCompute

When computing the default is expensive:

```typescript
import { err, getOrCompute } from "@deessejs/core";

// This is WRONG - computes default even when not needed
const bad = err("error").getOrElse(expensiveComputation());

// This is RIGHT - only computes when needed
const result = err("error").getOrCompute(() => expensiveComputation());
```

Example with database fallback:

```typescript
import { ok, err, getOrCompute } from "@deessejs/core";

const getUser = async (id: string) => {
  const cached = await getFromCache(id);

  return cached.getOrCompute(async () => {
    // Only hits database if not in cache
    return await database.users.findById(id);
  });
};
```

## Pattern Matching with match

When you need different behavior for success and failure:

```typescript
import { ok, err, match } from "@deessejs/core";

const displayUser = (userResult: Result<User, string>) => {
  const html = match(
    userResult,
    (user) => `<div class="user">${user.name}</div>`,
    (error) => `<div class="error">${error}</div>`
  );

  document.getElementById("app").innerHTML = html;
};
```

## Conditional Fallbacks

Chain multiple fallbacks:

```typescript
import { ok, err, Result, getOrElse, orElse } from "@deessejs/core";

// Try primary, then fallback
const getData = (key: string): Result<Data, string> => {
  const primary = fetchFromPrimary(key);

  return primary.orElse(() => fetchFromBackup(key));
};

// With getOrElse after chaining
const value = getData("settings")
  .orElse(() => getDefaultSettings())
  .getOrElse(DEFAULT_VALUE);
```

## Optional Values with Maybe

Same pattern for optional values:

```typescript
import { some, none, fromNullable, getOrElse } from "@deessejs/core";

interface User {
  name: string;
  settings?: {
    theme?: string;
    notifications?: boolean;
  };
}

const getTheme = (user: User): string => {
  return fromNullable(user.settings?.theme)
    .getOrElse("light");
};

const getNotifications = (user: User): boolean => {
  return fromNullable(user.settings?.notifications)
    .getOrElse(true); // Default to enabled
};
```

## When to Use Each Pattern

| Method | Use When |
|--------|----------|
| `getOrElse(default)` | Simple fallback value |
| `getOrCompute(fn)` | Expensive computation for default |
| `match(ok, err)` | Different behavior per case |
| `orElse(fn)` | Chain to another Result |

## Recap

- Use `getOrElse` for simple, immediate defaults
- Use `getOrCompute` for expensive computations that might not be needed
- Use `match` when success and failure need different handling
- Use `orElse` to chain fallback operations
