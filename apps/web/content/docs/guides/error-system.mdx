Traditional error handling with exceptions is hard to compose and easy to forget. The Error system provides structured errors inspired by Python's exception handling (PEP 654, PEP 678) that you can enrich, chain, and group.

## The Problem

JavaScript exceptions are hard to work with:

```typescript
try {
  await processOrder(order);
} catch (error) {
  // What type is error? Any?
  // How do I add context?
  // How do I chain errors?
  console.error(error);
  throw error; // Lost context!
}
```

Problems:
- No structured way to add context
- Hard to chain errors (the "cause")
- Can't group multiple errors together
- TypeScript doesn't know what errors are possible

## The Solution: Structured Errors

Create reusable error types with the `error()` function:

```typescript
import { error } from "@deessejs/core";

// Define error types like Python exception classes
const SizeError = error({
  name: "SizeError",
  args: {} as { current: number; wanted: number }
});

// Create errors with type safety
const sizeError = SizeError({ current: 3, wanted: 5 });
// Error { name: "SizeError", args: { current: 3, wanted: 5 }, notes: [], cause: null }
```

<Callout type="success">
  Errors are immutable and serializable - perfect for logging, API responses, or storage.
</Callout>

## Enriching Errors

Add notes to provide context:

```typescript
import { error } from "@deessejs/core";

const ValidationError = error({
  name: "ValidationError",
  args: {} as { field: string }
});

// Add notes for debugging
const err = ValidationError({ field: "email" })
  .addNotes("User submitted form at /checkout")
  .addNotes("Email validation failed regex check");

console.log(err.notes);
// ["User submitted form at /checkout", "Email validation failed regex check"]
```

## Chaining Errors

Track the cause chain like Python:

```typescript
import { error, err } from "@deessejs/core";

const NetworkError = error({
  name: "NetworkError",
  args: {} as { url: string }
});

const ValidationError = error({
  name: "ValidationError",
  args: {} as { field: string }
});

// Chain errors with .from()
const networkErr = NetworkError({ url: "/api/users" });
const validationErr = ValidationError({ field: "email" })
  .from(networkErr);  // Now validationErr.cause points to networkErr

console.log(validationErr.cause?.name); // "NetworkError"
```

## Grouping Errors

Handle multiple errors like Python's ExceptionGroup:

```typescript
import { error, exceptionGroup, isErrorGroup } from "@deessejs/core";

const ValidationError = error({
  name: "ValidationError",
  args: {} as { field: string }
});

const errors = [
  ValidationError({ field: "email" }),
  ValidationError({ field: "name" }),
  ValidationError({ field: "age" })
];

const group = exceptionGroup(errors);
// ErrorGroup { name: "ValidationError", exceptions: [...] }

// Check if grouped
if (isErrorGroup(group)) {
  console.log(`Found ${group.exceptions.length} validation errors`);
}
```

## Filtering Errors

Filter errors by name from a group:

```typescript
import { error, exceptionGroup, filterErrorsByName } from "@deessejs/core";

const NetworkError = error({
  name: "NetworkError",
  args: {} as { url: string }
});

const ValidationError = error({
  name: "ValidationError",
  args: {} as { field: string }
});

const group = exceptionGroup([
  NetworkError({ url: "/api/1" }),
  ValidationError({ field: "email" }),
  NetworkError({ url: "/api/2" }),
  ValidationError({ field: "name" })
]);

const networkErrors = filterErrorsByName(group, "NetworkError");
console.log(networkErrors.length); // 2
```

## Using with Result

The Error type integrates with Result:

```typescript
import { error, err, isErrWithError } from "@deessejs/core";

const SizeError = error({
  name: "SizeError",
  args: {} as { current: number; wanted: number }
});

// Errors work with Result
const result: Result<number, Error> = err(SizeError({ current: 3, wanted: 5 }));

// Type guard to check if Result contains an Error
if (isErrWithError(result)) {
  console.log(result.error.name); // "SizeError"
  console.log(result.error.args); // { current: 3, wanted: 5 }
}
```

## Zod Integration

Validate error arguments at runtime with Zod:

```typescript
import { error } from "@deessejs/core";
import { z } from "zod";

// Define with Zod schema for runtime validation
const SizeError = error({
  name: "SizeError",
  schema: z.object({
    current: z.number(),
    wanted: z.number()
  })
});

// Invalid args will throw a ValidationError
try {
  SizeError({ current: "three" }); // Wrong type!
} catch (e) {
  console.log(e.name); // "ValidationError"
  console.log(e.args.issues); // Zod validation issues
}
```

## The raise() Function

Convert errors for use in transformations:

```typescript
import { error, raise, ok, err, flatMap } from "@deessejs/core";

const NotFoundError = error({
  name: "NotFoundError",
  args: {} as { resource: string }
});

const findUser = (id: string) => {
  if (id === "123") return ok({ name: "Alice" });
  return err(NotFoundError({ resource: "user" }));
};

// Use raise() to throw in a flatMap chain
const result = ok("123")
  .flatMap(id => {
    const user = findUser(id);
    return user.isErr() ? raise(user) : ok(user.value);
  });
```

## Best Practices

1. **Define error types at module level** - Reusable, testable
2. **Use descriptive names** - `ValidationError` not `Err`
3. **Add rich args** - Include all debugging info
4. **Chain causes** - Preserve the error trail
5. **Group related errors** - Handle multiple failures at once
