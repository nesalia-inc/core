The `Result` type represents a value that can be either a success (`Ok`) or a failure (`Err`). It's the primary type for explicit error handling in your applications.

## Why Result?

Traditional error handling in TypeScript relies on exceptions:

```typescript
function divide(a: number, b: number): number {
  if (b === 0) throw new Error("Division by zero");
  return a / b;
}

try {
  const result = divide(10, 2);
  console.log(result * 2);
} catch (e) {
  console.error(e.message);
}
```

This approach has fundamental problems:

1. **Exceptions can escape** - Nothing guarantees you'll catch all errors
2. **No type safety** - The return type says `number`, but it can throw
3. **Nested code** - Error handling quickly becomes deeply nested
4. **Easy to forget** - It's simple to miss handling an error

<Callout type="warning">
  Did you notice the return type is `number`, but there's no way to know from the type signature that this function can throw?
</Callout>

## Enter Result: Explicit Error Handling

Result makes errors **explicit in your type signature**:

```typescript
import { ok, err, map, getOrElse } from "@deessejs/core";

const divide = (a: number, b: number): Result<number, string> => {
  if (b === 0) return err("Division by zero");
  return ok(a / b);
};

// TypeScript forces you to handle both cases
const result = divide(10, 2)
  .map(x => x * 2)     // Transform the value
  .getOrElse(0);       // Provide default if error

console.log(result); // 5
```

The error is now **explicit**—it's part of the type signature.

<Callout type="success">
  Now `Result<number, string>` clearly says: "this returns a number, but it might fail with a string error."
</Callout>

## Imperative vs Functional

Here's a validation function the traditional way:

<Tabs items={['Traditional (exceptions)', 'With Result']}>
<Tab value="Traditional (exceptions)">
```typescript
interface ValidationError {
  field: string;
  message: string;
}

function validateAge(age: number): number {
  if (age < 0) throw new Error("Age cannot be negative");
  if (age > 150) throw new Error("Invalid age");
  return age;
}

try {
  const age = validateAge(userInput);
  // use age...
} catch (e) {
  console.error(e.message);
  // But what if we need to show multiple errors?
}
```
</Tab>
<Tab value="With Result">
```typescript
import { ok, err, Result } from "@deessejs/core";

type ValidationError = { field: string; message: string };

const validateAge = (age: number): Result<number, ValidationError> => {
  if (age < 0) return err({ field: "age", message: "Age cannot be negative" });
  if (age > 150) return err({ field: "age", message: "Invalid age" });
  return ok(age);
};
```
</Tab>
</Tabs>

Now you can collect multiple errors, compose validations, and the type system tracks everything.

<Callout>
  Notice how the error type is explicit: `Result<number, ValidationError>`. The caller knows exactly what kind of error to expect.
</Callout>

## Overview

```typescript
type Result<T, E> = Ok<T> | Err<E>;
```

- **`Ok<T>`** - Represents a successful result containing a value of type `T`
- **`Err<E>`** - Represents a failure containing an error of type `E`

## Creating a Result

### ok - Create a Success

When an operation succeeds:

```typescript
import { ok, err, isOk, isErr } from "@deessejs/core";

const success: Result<number, string> = ok(42);
console.log(isOk(success)); // true
console.log(success.value);  // 42
```

But why wrap a simple value? Consider a validation function:

```typescript
const parseAge = (input: string): Result<number, string> => {
  const age = Number(input);
  if (isNaN(age)) return err("Invalid number");
  if (age < 0) return err("Age cannot be negative");
  return ok(age);
};
```

By returning `Result`, you make it **impossible** for callers to ignore the error case—TypeScript forces them to handle it.

### err - Create a Failure

When an operation fails:

```typescript
const failure: Result<number, string> = err("Something went wrong");
console.log(isErr(failure)); // true
console.log(failure.error);   // "Something went wrong"
```

## Transforming Results

### map - Transform the Value

When you want to transform the value without losing error context:

```typescript
import { ok, map, isOk } from "@deessejs/core";

const result = ok(10)
  .map(x => x * 2)
  .map(x => x + 1);

console.log(isOk(result)); // true
console.log(result.value); // 21
```

If it's an error, transformations are skipped:

```typescript
import { err, map } from "@deessejs/core";

const failed = err("error")
  .map(x => x * 2);

failed.error; // "error" - transformation was skipped
```

<Callout>
  Think of `map` as an assembly line: the value passes through transformations, but if something went wrong (error), the transformations are simply skipped.
</Callout>

### mapErr - Transform the Error

When you need to transform the error itself:

```typescript
import { err, mapErr } from "@deessejs/core";

const result = err("error")
  .mapErr(e => new Error(e));

console.log(result.error); // Error: error
```

This is useful for normalizing errors:

```typescript
const result = validate(input)
  .mapErr(errors => new ValidationError(errors));
```

### flatMap - Chain Operations

When your transformation returns a Result:

```typescript
import { ok, flatMap, err } from "@deessejs/core";

const parseNumber = (s: string): Result<number, string> => {
  const n = Number(s);
  return isNaN(n) ? err("Invalid number") : ok(n);
};

const divide = (a: number, b: number): Result<number, string> =>
  b === 0 ? err("Division by zero") : ok(a / b);

// Chain them together!
const result = ok("42")
  .flatMap(parseNumber)     // Ok(42) - not Ok(Ok(42))
  .flatMap(n => divide(n, 2))
  .map(n => n * 10);
// Ok(210)
```

<Callout type="warning">
  Without `flatMap`, you'd get nested Results: `Result<Result<Result<number>, string>, string>`—a nightmare!
</Callout>

## Handling Results

### getOrElse - Get Value or Default

For simple cases where you want a fallback:

```typescript
import { ok, err, getOrElse } from "@deessejs/core";

const success = ok(10).getOrElse(0);  // 10
const failure = err("e").getOrElse(0); // 0
```

### getOrCompute - Compute Default Lazily

When computing the default is expensive:

```typescript
import { err, getOrCompute } from "@deessejs/core";

const result = err("error").getOrCompute(() => {
  console.log("Computing default...");
  return 42;
});
// Logs: "Computing default..."
```

### match - Pattern Matching

For exhaustive handling that forces you to deal with all cases:

```typescript
import { ok, err, match } from "@deessejs/core";

const result = ok(42);
const message = match(
  result,
  (value) => `Success: ${value}`,
  (error) => `Error: ${error}`
);
// "Success: 42"
```

<Callout type="success">
  Why is this powerful? TypeScript will warn you if you add new error types but forget to handle them!
</Callout>

### tap - Side Effects

When you want to do something with the value without changing it:

```typescript
import { ok, tap } from "@deessejs/core";

ok(42)
  .tap(value => console.log("Got value:", value))
  .tap(value => console.log("Doubled:", value * 2));
// Logs: "Got value: 42"
// Logs: "Doubled: 84"
```

### tapErr - Side Effects on Error

When you want to handle errors with side effects:

```typescript
import { err, tapErr } from "@deessejs/core";

err("error")
  .tapErr(e => console.error("Failed:", e));
// Logs: "Failed: error"
```

## Type Guards

### isOk / isErr

Check what's inside without extracting:

```typescript
import { ok, err, isOk, isErr } from "@deessejs/core";

const result = ok(42);

if (isOk(result)) {
  // TypeScript knows result is Ok<number>
  console.log(result.value);
}

if (isErr(result)) {
  // TypeScript knows result is Err<never> here
  console.log(result.error);
}
```

## Converting to Other Types

Sometimes you need to transform a Result into another type to integrate with different parts of your codebase. Here's when and why you'd use each conversion.

### toMaybe - When You Don't Need Error Details

Use this when:
- You started with a Result but later realized you don't care about the specific error
- You're passing the result to code that expects a Maybe

Imagine you're validating user input at the API boundary, but deep in your business logic you just want to know "do we have a valid value or not":

```typescript
import { ok, err, toMaybe, isSome } from "@deessejs/core";

// Validation returns Result with detailed errors
const validateUserName = (input: string): Result<UserName, ValidationError> => {
  if (input.length < 3) return err({ field: "username", message: "Too short" });
  if (input.length > 20) return err({ field: "username", message: "Too long" });
  return ok(input as UserName);
};

// But downstream code just wants to know if it's valid
const maybeName = toMaybe(validateUserName(input));

// Now it's Some("john") or None - no error details needed
if (isSome(maybeName)) {
  createUserProfile(maybeName.value);
}
```

<Callout>
  When you convert to Maybe, you lose the error type information. Use this only when the specific error doesn't matter downstream.
</Callout>

### toOutcome - When You Need Rich Error Tracking

Use this when:
- You're building a system that needs to track both business errors and system exceptions
- You want to add cause information to errors
- You're integrating with logging systems that expect the Outcome format

The same error becomes more informative:

```typescript
import { ok, err, toOutcome, isSuccess, isCause, isException } from "@deessejs/core";

// Convert Result to Outcome for richer error tracking
const outcome = toOutcome(validateUserName(input));

// Now you have access to Success, Cause, or Exception variants
if (isSuccess(outcome)) {
  console.log("Valid user:", outcome.value);
} else if (isCause(outcome)) {
  // Track this for business metrics
  analytics.track("validation_failed", {
    field: outcome.cause.name,
    message: outcome.cause.message
  });
} else if (isException(outcome)) {
  // Log system errors differently
  logger.error(outcome.exception);
}
```

<Callout type="info">
  Outcome gives you three states: Success (happy path), Cause (business errors you expected), and Exception (unexpected system errors). This distinction helps you decide how to handle each case.
</Callout>

## When to Use Result

Use Result when:

- An operation can fail in predictable ways
- You need typed errors (not just `Error`)
- You want to compose operations that might fail
- You want TypeScript to force error handling
